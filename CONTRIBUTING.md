# Contributing to Shuttlecock

Thank you for your interest in contributing to Shuttlecock.

Shuttlecock is a systems-level project. Contributions are expected to be thoughtful, technically sound, and aligned with the projectâ€™s goals.

---

## Who Should Contribute

Shuttlecock is best suited for contributors interested in:
- Systems programming
- Compiler or runtime development
- GPU and parallel computing
- Performance engineering
- Infrastructure-level software

You do not need to be an expert, but you should be willing to learn and work carefully.

---

## Ways to Contribute

### 1. Code Contributions
You can contribute to:
- Kernel interface and DSL
- Intermediate Representation (IR)
- Runtime and execution engine
- CPU backend
- CUDA backend
- ROCm backend
- Scheduler and memory planner
- Benchmarks and test harnesses

Each subsystem is designed to be independently ownable.

---

### 2. Design and Architecture
- Propose changes via GitHub Discussions
- Back proposals with reasoning and examples
- Be open to feedback and revision

Design discussions should remain technical and evidence-based.

---

### 3. Documentation
- Improve clarity of existing documents
- Add diagrams and explanations
- Write onboarding material for new contributors

Clear documentation is considered a core contribution.

---

## Getting Started

1. Read `README.md` to understand the project scope
2. Read `MANIFESTO.md` to understand guiding principles
3. Browse GitHub Issues and Discussions
4. Comment on an issue you want to work on before starting

This avoids duplicated effort and misalignment.

---

## Contribution Workflow

1. Fork the repository
2. Create a feature branch
3. Make focused, well-scoped changes
4. Write or update tests where applicable
5. Submit a Pull Request (PR)

Each PR should:
- Solve one clearly defined problem
- Include a concise description
- Reference relevant issues or discussions

---

## Code Quality Expectations

- Prefer clarity over cleverness
- Avoid premature optimization
- Follow existing coding style
- Add comments where behavior is non-obvious
- Keep changes minimal and reviewable

Large or architectural changes must be discussed before implementation.

---

## Ownership and Responsibility

Contributors who consistently:
- Deliver high-quality work
- Participate in reviews
- Maintain subsystems

May be granted subsystem ownership or maintainer status.

Ownership includes responsibility for:
- Code quality
- Review of related PRs
- Long-term maintainability

---

## Communication Guidelines

- Be respectful and professional
- Focus on technical merit
- Avoid personal criticism
- Disagree with ideas, not people

Shuttlecock values long-term collaboration over short-term wins.

---

## Final Note

If you are unsure where to start, open a GitHub Discussion or ask for guidance.

Good contributions are welcomed.
Sustained contributions are recognized.
